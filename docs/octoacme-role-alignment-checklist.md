# OctoAcme — Role Alignment & Onboarding Checklist

## Purpose
Ensure every team member joining a project is properly onboarded, understands their role, and is integrated into the relevant workflows and communication channels. Use this template at project kick-off or when a new person joins mid-project.

---

## Section 1: Onboarding — All Roles

- [ ] Reviewed the [Project Management Overview](octoacme-project-management-overview.md)
- [ ] Reviewed the [Roles and Personas](octoacme-roles-and-personas.md) document and confirmed your role
- [ ] Added to the project board (e.g., GitHub Projects) and relevant communication channels
- [ ] Introduced to PM, PdM, and Engineering Lead
- [ ] Confirmed access to required tools (repository, CI/CD, design tools, observability dashboards, support systems)
- [ ] Reviewed the project charter / one-pager and understood the success metrics
- [ ] Confirmed working agreements (branching conventions, PR policy, meeting cadence)

---

## Section 2: Role-Specific Alignment

### Developers
- [ ] Engineering Lead has walked through the architecture and coding standards
- [ ] PR and branching conventions reviewed
- [ ] CI/CD pipeline access confirmed
- [ ] Assigned a first issue or task in the backlog

### Engineering Lead
- [ ] Reviewed existing architecture docs and technical decision records
- [ ] Aligned with PM on risk escalation path for technical issues
- [ ] Agreed on code review and design review cadence with developers

### UX Designer
- [ ] Access to design tools and asset repositories confirmed
- [ ] Reviewed existing user flows and any prior usability research
- [ ] Aligned with PdM on design feedback and review cadence
- [ ] Confirmed process for sharing designs and specs with developers

### DevOps/SRE
- [ ] Access to CI/CD pipelines and observability tooling confirmed
- [ ] Reviewed deployment and rollback runbooks
- [ ] Aligned with Engineering Lead on release planning process
- [ ] On-call schedule and incident escalation path confirmed

### Business Analyst
- [ ] Reviewed existing requirements documentation and backlog
- [ ] Aligned with PM and PdM on requirements intake process
- [ ] Confirmed UAT responsibilities and stakeholder contacts
- [ ] Access to stakeholder communication channels confirmed

### QA/Testing
- [ ] Test environments and tooling access confirmed
- [ ] Reviewed the Definition of Done and acceptance criteria standards
- [ ] Aligned with Engineering Lead on test coverage expectations
- [ ] Agreed on QA sign-off process before releases

### Support Lead
- [ ] Access to customer support tooling and feedback channels confirmed
- [ ] Agreed on process for escalating customer-impacting bugs to PM/PdM
- [ ] Added to release review and retrospective meetings
- [ ] Aligned on weekly support summary cadence with PM

### Product Manager (PdM)
- [ ] Backlog reviewed and prioritized for current sprint/milestone
- [ ] Roadmap and success metrics shared with delivery team
- [ ] Acceptance criteria process agreed upon with QA and developers
- [ ] Stakeholder communication plan confirmed with PM

### Project Manager (PM)
- [ ] Project charter created or updated
- [ ] Risk register initialized with known risks and owners
- [ ] Meeting cadence established (standups, weekly sync, demos)
- [ ] Escalation paths communicated to all team members

---

## Section 3: Communication & Collaboration

- [ ] All roles are represented in recurring meetings appropriate to their function
- [ ] Single source of truth for project status identified and shared
- [ ] Escalation paths from [Risk Management & Communication](octoacme-risks-and-communication.md) reviewed with the team
- [ ] Hand-off checklist from [Execution & Tracking](octoacme-execution-and-tracking.md) shared with relevant roles
